
# RentGuard 360 — Installation & Deployment (Single Document)

This is the single “source of truth” installation document for RentGuard 360.
It is written for a fresh AWS account and for a recipient who gets the deployment package + scripts and only needs to install the system.

## What you receive
You will receive one of the following:

### Option A (recommended): Deployment package
- `RentGuard360-Deployment.zip` (generated by our scripts)

### Option B: Git repository access
- The full source code repository + deployment scripts

## Deliverables for sections 9–10 (submission checklist)
To fully cover the formal requirements “9–10” (source bundle + installation instructions), provide these deliverables:

1) **Source bundle (requirement 9)**
- `RentGuard360-Source.zip` (a ZIP containing *all project source files* used to build/deploy the system)
- OR a repository link (GitHub/Azure DevOps/GitLab) **plus** a ZIP export of the sources (recommended to avoid access issues)

Minimum required content inside the source bundle:
- All Lambda source files: `backend/lambdas/*.py` (and any `.json` / `.txt` used by the Lambdas)
- API interface definition (Swagger/OpenAPI): `backend/api-gateway/RentGuardAPI-prod-swagger-apigateway.json`
- Infrastructure template: `infrastructure/cloudformation.yaml`
- Frontend source: `frontend/src`, `frontend/package.json`, `frontend/vite.config.js`, etc.

2) **Deployment bundle (optional but recommended)**
- `RentGuard360-Deployment.zip` (generated by `infrastructure/create-deployment-package.bat`)

Important: `RentGuard360-Deployment.zip` is designed for deployment convenience. It includes `lambdas.zip` (packaged code) but does **not** replace the full “source bundle” requirement by itself.

3) **Installation instructions (requirement 10)**
- This document: `DEPLOYMENT_INSTRUCTIONS.md`
- If you provide a repository link, include it here under “Where to clone from”.

## Secrets policy (do NOT attach secrets)
Do not include real secrets in any ZIP or repository:
- `infrastructure/config.env`
- `frontend/.env`
- Any file containing `AZURE_DOC_KEY` or other keys/tokens

Provide only templates:
- `infrastructure/config.env.template`
- `frontend/.env.template`

The installer must fill secrets locally (or receive them via a secure channel separate from the source bundle).

## Installation (AWS CloudShell) — using the deployment ZIP

### 1) Upload and extract
In AWS CloudShell, upload `RentGuard360-Deployment.zip`, then:

```bash
rm -rf ~/infrastructure ~/frontend ~/backend ~/docs ~/DEPLOYMENT_INSTRUCTIONS.md 2>/dev/null || true
unzip -o RentGuard360-Deployment.zip
```

Note: CloudShell may show a warning about backslashes as path separators. This is expected and extraction still works.

### 2) Create / place `config.env`
You have two choices:

#### Option A: You uploaded `config.env` already
Move it into the right folder:

```bash
mv -f config.env infrastructure/
```

#### Option B: Create it from the template in CloudShell

```bash
cd infrastructure
cp -f config.env.template config.env
nano config.env
```

Fill these required values:
- `AZURE_DOC_ENDPOINT`
- `AZURE_DOC_KEY`
- `SENDER_EMAIL`

Default values you normally keep:
- `STACK_NAME="RentGuard360"`
- `ENVIRONMENT="prod"`
- `AWS_REGION="us-east-1"`

### 2.1) Two isolated stacks in the same AWS account (recommended for testing)
This project supports deploying a completely separate test stack in the SAME AWS account.

Use these values in `infrastructure/config.env`:
- Main stack:
	- `STACK_NAME="RentGuard360"`
	- `NAME_SUFFIX=""`
- Isolated test stack:
	- `STACK_NAME="RentGuard360-Test"`
	- `NAME_SUFFIX="-test"`

Safety behavior:
- The deployment scripts require `STACK_NAME` to exist in `config.env` (to avoid accidental main-stack updates).
- Deploying the main stack (`STACK_NAME=RentGuard360` with empty `NAME_SUFFIX`) requires typing `DEPLOY_MAIN` to continue.

### 3) Run the deployment script

```bash
cd ~/infrastructure
chmod +x deploy-cloudshell.sh
./deploy-cloudshell.sh
```

The script automatically handles Windows line endings (CRLF) in `config.env` if needed.

### 4) Packaging detail (Lambda filenames)
During packaging we convert Lambda filenames from `-` to `_` inside `lambdas.zip` (Python import-safe).
This is why Lambda handlers in CloudFormation use underscore module names.

### 5) How the deployment works (important)
- The scripts upload:
	- `infrastructure/dist/lambdas.zip`
	- `infrastructure/cloudformation.yaml`
	to a per-account deployment bucket named `rentguard-deployment-<AccountId>`.
- CloudFormation is deployed using `TemplateURL` (not `TemplateBody`) to avoid the 51,200-byte CLI size limit.

## Frontend deployment
The CloudShell deploy script uploads the frontend only if a local `frontend/dist` exists (it usually does not in the ZIP).
Common approach: build locally and upload.

### 1) Create `frontend/.env` locally (do not commit)
On your machine:

```bash
cd frontend
cp .env.template .env
```

Get values from CloudFormation outputs:

```bash
STACK_NAME=RentGuard360

API_URL=$(aws cloudformation describe-stacks --stack-name "$STACK_NAME" --query "Stacks[0].Outputs[?OutputKey=='ApiUrl'].OutputValue" --output text)
USER_POOL_ID=$(aws cloudformation describe-stacks --stack-name "$STACK_NAME" --query "Stacks[0].Outputs[?OutputKey=='UserPoolId'].OutputValue" --output text)
USER_POOL_CLIENT_ID=$(aws cloudformation describe-stacks --stack-name "$STACK_NAME" --query "Stacks[0].Outputs[?OutputKey=='UserPoolClientId'].OutputValue" --output text)
CONTRACTS_BUCKET=$(aws cloudformation describe-stacks --stack-name "$STACK_NAME" --query "Stacks[0].Outputs[?OutputKey=='ContractsBucketName'].OutputValue" --output text)
CHECK_USER_API_KEY_ID=$(aws cloudformation list-stack-resources --stack-name "$STACK_NAME" --query "StackResourceSummaries[?LogicalResourceId=='CheckUserApiKey'].PhysicalResourceId" --output text)
CHECK_USER_API_KEY=$(aws apigateway get-api-key --api-key "$CHECK_USER_API_KEY_ID" --include-value --query "value" --output text)

echo "VITE_API_ENDPOINT=$API_URL" > .env
echo "VITE_CHECK_USER_API_KEY=$CHECK_USER_API_KEY" >> .env
echo "VITE_USER_POOL_ID=$USER_POOL_ID" >> .env
echo "VITE_USER_POOL_CLIENT_ID=$USER_POOL_CLIENT_ID" >> .env
echo "VITE_AWS_REGION=us-east-1" >> .env
echo "VITE_S3_BUCKET=$CONTRACTS_BUCKET" >> .env
```

Security note:
- `VITE_CHECK_USER_API_KEY` is intended for API Gateway usage-plan throttling and abuse control for the public `/auth/check-user` endpoint.
- Rotate this key if exposed and redeploy/rebuild frontend.

### 2) Build and upload

```bash
npm install
npm run build

FRONTEND_BUCKET=$(aws cloudformation describe-stacks --stack-name "$STACK_NAME" --query "Stacks[0].Outputs[?OutputKey=='FrontendBucketName'].OutputValue" --output text)
aws s3 sync dist/ "s3://$FRONTEND_BUCKET/" --delete
```

### 3) Invalidate CloudFront (recommended)

```bash
DIST_ID=$(aws cloudformation describe-stacks --stack-name "$STACK_NAME" --query "Stacks[0].Outputs[?OutputKey=='CloudFrontDistributionId'].OutputValue" --output text)
aws cloudfront create-invalidation --distribution-id "$DIST_ID" --paths "/*"
```

## Create an admin user (Cognito)

```bash
STACK_NAME=RentGuard360
USER_POOL_ID=$(aws cloudformation describe-stacks --stack-name "$STACK_NAME" --query "Stacks[0].Outputs[?OutputKey=='UserPoolId'].OutputValue" --output text)

aws cognito-idp admin-create-user \
	--user-pool-id "$USER_POOL_ID" \
	--username your-email@example.com \
	--user-attributes Name=email,Value=your-email@example.com Name=email_verified,Value=true \
	--message-action SUPPRESS \
	--temporary-password "TempPass123!"

# Set a permanent password (avoids relying on Cognito/SES emails)
aws cognito-idp admin-set-user-password \
	--user-pool-id "$USER_POOL_ID" \
	--username your-email@example.com \
	--password "YourPassword123!" \
	--permanent

aws cognito-idp admin-add-user-to-group \
	--user-pool-id "$USER_POOL_ID" \
	--username your-email@example.com \
	--group-name Admins
```

### Notes about SES Sandbox

If SES is in sandbox mode, Cognito may fail to send emails to arbitrary addresses.
The commands above use `--message-action SUPPRESS` + `admin-set-user-password --permanent` so admin creation does not rely on email delivery.


## Verification checklist (post-deploy)

### 1) Stack status and outputs

```bash
STACK_NAME=RentGuard360
aws cloudformation describe-stacks --stack-name "$STACK_NAME" --query "Stacks[0].StackStatus" --output text
aws cloudformation describe-stacks --stack-name "$STACK_NAME" --query "Stacks[0].Outputs" --output table
```

### 2) Verify API Gateway `GET /upload`
The frontend expects `GET /upload?fileName=...`.

```bash
STACK_NAME=RentGuard360
REGION=us-east-1

API_ID=$(aws cloudformation list-stack-resources --stack-name "$STACK_NAME" --region "$REGION" \
	--query "StackResourceSummaries[?ResourceType=='AWS::ApiGateway::RestApi'] | [0].PhysicalResourceId" --output text)

UPLOAD_ID=$(aws apigateway get-resources --rest-api-id "$API_ID" --region "$REGION" \
	--query "items[?path=='/upload'] | [0].id" --output text)

aws apigateway get-method --rest-api-id "$API_ID" --region "$REGION" --resource-id "$UPLOAD_ID" --http-method GET \
	--query "{httpMethod:httpMethod,auth:authorizationType}" --output table
```

### 3) Verify API Gateway `GET /auth/check-user` (+ API key required)

```bash
STACK_NAME=RentGuard360
REGION=us-east-1

API_ID=$(aws cloudformation list-stack-resources --stack-name "$STACK_NAME" --region "$REGION" \
	--query "StackResourceSummaries[?ResourceType=='AWS::ApiGateway::RestApi'] | [0].PhysicalResourceId" --output text)

CHECK_USER_ID=$(aws apigateway get-resources --rest-api-id "$API_ID" --region "$REGION" \
	--query "items[?path=='/auth/check-user'] | [0].id" --output text)

aws apigateway get-method --rest-api-id "$API_ID" --region "$REGION" --resource-id "$CHECK_USER_ID" --http-method GET \
	--query "{httpMethod:httpMethod,auth:authorizationType,apiKeyRequired:apiKeyRequired}" --output table
```

Expected:
- `auth` should be `NONE`
- `apiKeyRequired` should be `true`

### 4) CloudFront is deployed

```bash
STACK_NAME=RentGuard360
DIST_ID=$(aws cloudformation describe-stacks --stack-name "$STACK_NAME" --query "Stacks[0].Outputs[?OutputKey=='CloudFrontDistributionId'].OutputValue" --output text)
aws cloudfront get-distribution --id "$DIST_ID" --query "Distribution.Status" --output text
```

## Troubleshooting

### CloudShell shows `$'\r': command not found`
This means `config.env` has Windows CRLF line endings.
The deployment script auto-fixes this, but if needed you can run:

```bash
sed -i 's/\r$//' config.env
```

### SES is not sending emails
- Verify your sender email in AWS SES.
- If SES is in sandbox mode, recipients must also be verified.

### Upload fails in browser with a CORS error
If the browser console shows something like:
`blocked by CORS policy` when uploading to S3 with a presigned URL:
- Make sure your stack was deployed/updated using the latest `cloudformation.yaml` from the package/repo.
- The `ContractsBucket` CORS rules are managed by CloudFormation and should allow uploads from your CloudFront domain.

### Reset Password fails with `CustomMessage failed ... Unable to import module 'custom_message'`
This means your Cognito User Pool has a **Custom Message trigger** configured, but the Lambda is missing or the handler name is wrong.

This project does **not** require a Custom Message trigger for password reset. The HTML is set via the User Pool **Message templates** (CloudFormation `VerificationMessageTemplate`) and should be used for both verification and password reset codes.

Fix options:
- Preferred: redeploy / update the CloudFormation stack (it will remove the `CustomMessage` trigger wiring).
- Quick manual fix: AWS Console → Cognito → User Pools → your pool → **Triggers** → set **Custom message** to **None** (or remove it), then retry “Forgot password”.
